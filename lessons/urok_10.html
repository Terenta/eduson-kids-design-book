Как это работает

Streamlit перезапускает весь скрипт

Главное правило урока: при каждом действии пользователя — ввод текста, клик по кнопке — Streamlit прогоняет весь app.py заново, сверху вниз.

Отсюда простота. Вы просто пишете код сверху вниз, как обычный скрипт. Никаких обработчиков событий и обновлений вручную — Streamlit сам перерисовывает страницу.
Отсюда проблема. Раз скрипт запускается заново — обычные переменные обнуляются. История переписки сотрётся при каждом новом вопросе. Это чиним на следующем слайде.
Запомните одну фразу: Streamlit перезапускает весь app.py на каждый ввод и клик. Из неё следует всё остальное.
Как это работает

Память диалога — st.session_state

Обычная переменная на каждом перезапуске обнуляется. Чтобы переписка не терялась, историю храним в особом месте — st.session_state.

Обычная переменная

Создаётся заново при каждом запуске скрипта. После вопроса всё, что в ней было, пропадает.

st.session_state

Хранилище, которое живёт между перезапусками. Здесь живёт история чата — список st.session_state.messages.
st.session_state.messages — это список, как entries в дневнике урока 7. Только он живёт не в файле, а в памяти сессии, пока открыта вкладка.
Новая роль нейросети

Нейросеть отвечает в браузере

На уроке 5 DeepSeek был советчиком, на уроке 9 жил в Telegram-боте. Сегодня движок тот же, но ответ рисуется прямо на веб-странице.

Системная роль

«Ты — полезный помощник. Отвечай понятно и по делу». Вопрос из поля уходит в сеть, ответ возвращается текстом.
Всю работу с сетью прячем в функцию ask_deepseek(question): она шлёт запрос к DeepSeek и возвращает готовый текст ответа, который сразу показываем на странице.

Типичные ситуации с нейросетью

ModuleNotFoundError
Не установлены streamlit или requests. В терминале: pip install streamlit requests python-dotenv — и запускайте снова.
401 Unauthorized
Пустой ответ или ошибка от DeepSeek — ключ не подставлен. Проверьте, что API_KEY на месте: локально в .env, а на Streamlit Cloud — в разделе Secrets.

Правка 1 · отвечает нейросеть

Вопрос уходит в сеть, ответ — на странице

Первый шаг: поле больше не эхо. Вопрос уходит в DeepSeek API, а ответ появляется прямо на веб-странице.

→ В тот же чат
Подключи DeepSeek через библиотеку requests (requests.post на https://api.deepseek.com/chat/completions, модель deepseek-chat): вопрос из поля отправляется нейросети, ответ показывается на странице вместо эха. Ключ читай из переменной API_KEY. Не используй библиотеку openai. Верни весь файл.
Разбор кода

Запрос к DeepSeek через requests

PYTHON
import requests def ask_deepseek(question): headers = {"Authorization": f"Bearer {API_KEY}"} data = { "model": "deepseek-chat", "messages": [ {"role": "system", "content": "Ты полезный помощник"}, {"role": "user", "content": question}, ], } url = "https://api.deepseek.com/chat/completions" resp = requests.post(url, headers=headers, json=data) return resp.json()["choices"][0]["message"]["content"]
Тот же движок, что в советчике и боте, но обращаемся напрямую через requests.post: url https://api.deepseek.com/chat/completions, модель deepseek-chat. Ответ берём из choices[0].message.content. Библиотеку openai не ставим — это тема урока 11.
Правка 2 · чат-история

Настоящий чат, который помнит

Теперь поле превращается в чат: сообщения показываются пузырями, а вся переписка держится в памяти и не стирается.

→ В тот же чат
Сделай интерфейс чатом: используй st.chat_input и st.chat_message, храни переписку в st.session_state.messages, показывай всю историю. Верни весь файл.
Разбор кода

История чата живёт в session_state

PYTHON
if "messages" not in st.session_state: # память между перезапусками st.session_state.messages = [] for m in st.session_state.messages: # показать историю st.chat_message(m["role"]).write(m["content"]) question = st.chat_input("Спроси что-нибудь") if question: st.session_state.messages.append({"role": "user", "content": question}) answer = ask_deepseek(question) # запрос к DeepSeek через requests st.session_state.messages.append({"role": "assistant", "content": answer}) st.rerun()

Каждое сообщение — словарь с полями role и content. Список messages — как entries из дневника, только живёт в памяти сессии. st.rerun перерисовывает страницу с новой репликой.

Разбор кода

Ключ — в Secrets, не в коде

Файл app.py попадает на публичный GitHub — всё, что в нём написано, увидят посторонние. Ключ, попавший в код, увидят все. Читаем его в ту же переменную API_KEY, что и в ask_deepseek:

PYTHON
import os import streamlit as st # в облаке — из Secrets, локально — из .env API_KEY = st.secrets.get("DEEPSEEK_KEY") or os.getenv("DEEPSEEK_KEY")

Сам ключ живёт в панели Streamlit Cloud, в разделе Secrets, в формате toml:

TOML
DEEPSEEK_KEY = "sk-..."
Никогда не пишите API_KEY = "sk-..." прямо в app.py. На хостинге секрет кладут в Secrets и читают через st.secrets["DEEPSEEK_KEY"] — тогда на GitHub попадает только код, без ключа.

KeyError значит: секрет не добавлен в Secrets на Cloud.

Если застряли

  • Скопируйте traceback целиком в чат DeepSeek — попросите объяснить и починить
  • ModuleNotFoundError на Cloud — забыли requirements.txt со списком библиотек
  • Приложение «спит» на бесплатном тарифе — нажмите кнопку «wake up» и подождите

Привычки работы со Streamlit

01 · Streamlit = Python как сайт
Виджеты st.title, st.chat_input, st.write рисуют страницу — HTML писать не нужно.
02 · Память — в session_state
Streamlit перезапускает весь скрипт. Историю чата храним в st.session_state, иначе она стирается.
03 · Ключ — не в коде
Всегда в Secrets, читаем через st.secrets["DEEPSEEK_KEY"]. На GitHub попадает только код.
04 · requirements.txt
Список библиотек, чтобы Streamlit Cloud их поставил. Без него — ModuleNotFoundError.
05 · Публикация = ссылка
GitHub плюс Streamlit Cloud дают свою ссылку — сайт открывается с любого устройства.
