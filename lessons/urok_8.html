Новое понятие

Что такое API

API — это способ попросить данные у чужого сервера по понятным обеим сторонам правилам. Программа отправляет запрос с параметрами — сервер возвращает JSON-ответ.

Как заказ по меню: вы не идёте на кухню, а выбираете позицию по меню и получаете готовое блюдо. Программа не работает внутри OpenWeather: отправляет запрос по правилам и получает готовые данные о погоде.

JSON-ответ вам уже знаком — точно такой формат разбирали на уроке 7 в дневнике.

Как устроен запрос

Из чего состоит обращение к API

У запроса два куска: url — адрес сервиса погоды, и params — что именно спрашиваем.

PYTHON
url = "https://api.openweathermap.org/data/2.5/weather" params = {"q": city, "appid": OW_KEY, "units": "metric", "lang": "ru"} data = requests.get(url, params=params).json()
q
Город, о котором спрашиваем.
appid
Ключ — пропуск к сервису.
units=metric
Градусы Цельсия, а не Кельвины.
lang=ru
Описание погоды по-русски.
Прячем секреты

Что такое файл .env

Ключ — это пароль к сервису. Держать его строкой прямо в коде опасно: выложите код в интернет — ключ украдут и исчерпают ваш лимит.

Ключ строкой в коде

appid = "a1b2c3..." прямо в файле. Виден всем, кто увидит код.

Так делать небезопасно.

Ключ в файле .env

Секреты лежат рядом со скриптом, в код не попадают. load_dotenv() и os.getenv(...) читают их из .env.

.env добавляем в .gitignore — в интернет он не уходит.

В коде ключа строкой быть не должно — только чтение из .env.

Как это работает

Ответ сервера — знакомый JSON

OpenWeather присылает JSON — тот самый формат из урока 7. Это словарь: внутри вложены другие словари и список. Достаём из него нужные поля.

JSON
{ "main": { "temp": 23.4, "humidity": 60 }, "weather": [ { "description": "ясно" } ], "name": "Москва" }

Температура

Лежит внутри словаря main: берём data["main"]["temp"].

Описание

weather — это список, поэтому сначала [0]: data["weather"][0]["description"].

Разбор кода

Как программа спрашивает погоду

PYTHON
url = "https://api.openweathermap.org/data/2.5/weather" params = {"q": city, "appid": OW_KEY, "units": "metric", "lang": "ru"} data = requests.get(url, params=params).json() # ответ сервера как словарь temp = data["main"]["temp"] desc = data["weather"][0]["description"]

Параметры складываем в словарь params, а requests.get сам собирает из них адрес — вручную URL не склеиваем. .json() превращает ответ сервера в обычный словарь Python, и мы достаём из него нужные поля.

Правка 2 · ключи в .env

Переносим ключи в .env

Главный шаг к безопасности: секретные ключи переходят из кода в отдельный файл .env — как пароль, который не пишут в открытом тексте.

→ В тот же чат
Спрячь ключи. Создай рядом файл .env со строками OPENWEATHER_KEY=... и DEEPSEEK_KEY=..., а в коде читай их через библиотеку python-dotenv (load_dotenv, os.getenv). В коде не должно остаться ключей строкой. Верни весь файл целиком.
Разбор кода

Как ключи попадают в код из файла

PYTHON
from dotenv import load_dotenv import os load_dotenv() # прочитали файл .env OW_KEY = os.getenv("OPENWEATHER_KEY") # достали ключ по имени DS_KEY = os.getenv("DEEPSEEK_KEY")
load_dotenv() читает файл .env, а os.getenv достаёт ключ по имени. Теперь ключа строкой в коде нет — код можно показывать и выкладывать, а .env добавляют в .gitignore и держат только у себя.
Полезное знание

Один параметр для привычной температуры

Без units=metric

OpenWeather отдаёт градусы Кельвина: temp = 296.

На улице +23, но без параметра приходит 296.

С units=metric, lang=ru

Приходят привычные градусы Цельсия: temp = 23.

А lang=ru делает описание русским.

Кельвины — шкала физиков: 296 K − 273 ≈ 23 °C. С параметром units=metric температура приходит в привычном виде.

Нейросеть комментирует погоду

Программа берёт текущую погоду и передаёт её нейросети, чтобы получить короткий комментарий к данным.

→ В тот же чат
Отправь погоду в DeepSeek с системной ролью «Ты весёлый ведущий прогноза погоды, прокомментируй погоду одним бодрым абзацем» и распечатай ответ нейросети под данными о погоде. Ключ DeepSeek бери из .env. Верни весь файл целиком.
Разбор кода

Передаём погоду нейросети

PYTHON
data = { "model": "deepseek-chat", "messages": [ {"role": "system", "content": "Ты весёлый ведущий прогноза погоды, прокомментируй погоду одним бодрым абзацем."}, {"role": "user", "content": f"Город {city}: {temp} градусов, {desc}."}, ], } response = requests.post(url, headers=headers, json=data) answer = response.json()["choices"][0]["message"]["content"]

Системная роль задаёт стиль ответа — ведущий прогноза. В user-сообщение подставляем текущую погоду: город, температуру и описание. Нейросеть возвращает короткий комментарий, и мы печатаем его под данными.

Правка 4 · совет и цикл

Спрашиваем город по кругу

Разовый запуск превращаем в приложение: город спрашивается снова и снова, а к каждой погоде нейросеть советует, что надеть.

→ В тот же чат
Спрашивай город по кругу, пока не введут «выход». К каждой погоде проси у нейросети короткий совет, что надеть. Верни весь файл целиком.
Разбор кода

Круг вопросов и совет от нейросети

PYTHON
while True: city = input("Город (или «выход»): ") if city == "выход": break # ... запросили погоду и комментарий ... advice = ask_ai(f"Погода {temp} градусов, {desc}. Что надеть? Коротко.") print(advice)
Цикл while True спрашивает город снова и снова, пока не введут «выход» — тогда break выходит. Для совета делаем второй запрос к нейросети: погода уходит к DeepSeek уже с вопросом, что надеть.
Правка 5 · цвет и ошибки

Цвет и защита от ошибок

Финальная правка: делаем вывод цветным и учим приложение не падать на несуществующем городе.

→ В тот же чат
Финальная правка. Раскрась вывод ANSI-кодами, без сторонних библиотек: - город — голубым - температуру — жёлтым - комментарий нейросети — зелёным И сделай приложение устойчивым: если такого города нет (OpenWeather вернул 404) — вежливо скажи «город не найден» и продолжи работу, без падения программы. Верни весь файл.
Разбор кода

Проверяем ответ сервера

OpenWeather в каждом ответе кладёт числовой код в поле cod — прошёл запрос или нет. Проверяем его до того, как достаём температуру:

PYTHON
data = requests.get(url, params=params).json() if str(data["cod"]) != "200": # 200 — всё хорошо print("Город не найден") # 404 — города нет else: temp = data["main"]["temp"] # только теперь читаем погоду desc = data["weather"][0]["description"]
Если полезть в data["main"] сразу, а города нет — программа упадёт с KeyError. Сначала проверяем cod, потом читаем данные. Это общее правило для любого API.

Отладка

Читаем ошибки API

Traceback — снизу вверх. Последняя строка — что случилось, строкой выше — где.

ТЕРМИНАЛ
Traceback (most recent call last): File "weather.py", line 12, in <module> temp = data["main"]["temp"] KeyError: 'main'

KeyError: 'main' значит: в ответе нет погоды — скорее всего города не нашли или ключ не активен.

Если застряли

  • Скопируйте traceback целиком в чат DeepSeek — попросите объяснить и починить
  • Код 401 — ключ ещё не активирован, подождите; 404 — проверьте название города

Правила работы с API

01 · Запрос с параметрами
Программа шлёт запрос на url с params и получает JSON-ответ — данные от чужого сервера.
02 · units=metric
Всегда для погоды, иначе OpenWeather отдаёт температуру в Кельвинах, а не в градусах Цельсия.
03 · Секреты — в .env
Ключи читаем через load_dotenv и os.getenv. В коде ключа строкой быть не должно.
04 · .env — в .gitignore
Файл с ключами не выкладываем в публичный репозиторий: украдут — придётся перевыпускать.
05 · Проверяй код ответа
Смотрим cod: 200 — данные есть, 404 — города нет. Читаем данные только после проверки.

  • Несуществующий город — «город не найден», без падения
